# Decisions — Scrollback

Append-only. Architectural decisions and non-obvious gotchas. Format: date — title / What / Why / Rejected (or: If you change it). Seeded from tech-spec.md at scaffold time.

## 2026-07-11 — Product name: Scrollback
**What:** The product is named Scrollback ("scroll back through your day"; the terminal scrollback buffer is the text history of the screen).
**Why:** Distinctive, English, memorable for the dev audience. `getscrollback.com` was unregistered on 2026-07-10; the only prior use is a defunct 2014 community-chat startup in an unrelated category. Doc.md's earlier candidates were dead: "Engram" is a $98M-funded AI-memory startup; "Mnemo" has ~6 claimants.
**If you change it:** rename before the trademark screen and before acquiring scrollback.app/.ai (both parked). Trademark clearance (US/EU/IN classes 9/42) is still an open pre-launch TODO.

## 2026-07-10 — Swift end-to-end, three-process trust topology
**What:** Native Swift app; capture/index daemon (`scrollbackd`) with zero network code; separate `scrollback-courier` for all egress; thin Node MCP proxy (`.mcpb`).
**Why:** OS APIs are Swift-first; the process split *is* the provable-privacy claim (the process that sees your screen has no network code path — verifiable with LuLu/Little Snitch); solo-dev velocity.
**Rejected:** Rust core (FFI tax; cross-platform is a non-goal), Electron/Tauri (a web runtime in an always-on trust product is disqualifying), monolith (unverifiable egress claim).

## 2026-07-10 — EmbeddingGemma-300m Q4_0 on bundled llama.cpp, 512d Matryoshka
**What:** Primary on-device embedder; Qwen3-Embedding-0.6B as quality tier / legal fallback; Granite R2 311M tracked for v2; model downloads on first launch.
**Why:** Purpose-built for always-on/battery workloads, <200MB RAM, broadest runtime support; a statically bundled llama.cpp is the verified consumer shipping pattern (MacWhisper/Jan) and keeps the DMG <50MB.
**Rejected:** Ollama dependency (consumer-hostile; it's a dev-tool pattern), jina-v5 (CC BY-NC — cannot bundle), LFM2.5 ($10M-revenue license poison pill), NLContextualEmbedding as primary (unbenchmarked quality).
**Gotcha:** EmbeddingGemma is under Gemma Terms, NOT Apache — requires a notice file + Prohibited-Use flow-down in ToS, and Google retains a remote-restriction clause. If legal rejects it, Qwen3-0.6B (Apache 2.0) is the drop-in primary. Every chunk row stores `model_id`+`dim` so re-embedding is lazy.

## 2026-07-10 — WhisperKit STT, macOS 15+ floor, FluidAudio opt-in diarization
**What:** Single STT path at v1 (WhisperKit/ANE); Apple Silicon only; Apple SpeechAnalyzer adopted opportunistically on macOS 26+ later; ephemeral speaker labels, never voiceprints.
**Why:** One code path for a solo dev; the orphaned Rewind base skews to Sequoia; BIPA litigation over voiceprints is live.
**Rejected:** SpeechAnalyzer-only (a macOS-26 floor cuts the orphaned base), whisper.cpp primary (worst battery — Rewind's mistake).

## 2026-07-10 — SQLite + FTS5 + sqlite-vec behind RetrievalStore; RRF hybrid; weekly shards; SQLCipher + Secure-Enclave key
**What:** One encrypted embedded store; multi-list RRF (BM25 + vector + recency); episode segmentation; MinHash dedup; DB key wrapped by a Secure-Enclave P-256 key, gated by `.biometryCurrentSet`; purge = drop the weekly shard file.
**Why:** Only stack keeping text/FTS/vectors/metadata in one transactional encrypted file at our scale; SE key custody defeats the verified AMOS-class infostealer playbook (stolen keychain + phished password still can't unwrap it); shards make TTL/purge instant and provable.
**Rejected:** sqlite-vector (Elastic License → paid for us), hnswlib (unmaintained since 2023), DuckDB VSS (experimental persistence, corruption risk), LanceDB/ObjectBox (adds a second storage system), `.userPresence` key flag (falls back to the phishable login password).
**Gotcha:** sqlite-vec is pre-v1, single-maintainer (had a ~12.5-month 2025 gap). It sits behind a `RetrievalStore` protocol so the store is swappable — hedges are usearch (Apache, Swift bindings) and SQLite core's Vec1 once it ships 1.0 with acceptable licensing. Re-evaluate at ~1M vectors. macOS has NO VBS-style compute enclave — we isolate the key, not the compute; state that to auditors.

## 2026-07-10 — Filing agents: quarantined extraction → schema-validated fields → approve-before-commit → append-only idempotent Notion writes; read-only MCP surface
**What:** CaMeL-style split (extractor sees ambient text and emits schema JSON only; composer never sees raw text); draft-then-approve queue with earned autonomy; client-side `external_key` idempotency; write ledger + archive-undo; no agent-emitted URLs/images; Claude gets read-only MCP tools (filing is scheduler-initiated, not Claude-invoked, in v1).
**Why:** Scrollback holds the full "lethal trifecta" (private memory + untrusted captured content + a Notion-writing agent); the PromptArmor Notion-AI exfiltration attack is the exact precedent; every shipping analog (Highlight, Granola, LUCI) defaults human-in-the-loop; 68% of consumers won't trust unreviewed agent actions.
**Rejected:** silent auto-commit (trust suicide), prompt-level guardrails as security (verified insufficient — classifiers are probabilistic), Claude-invoked write tools in v1 (unnecessary blast radius).
**Gotcha:** the Notion API has no idempotency keys, so retried writes can duplicate — dedup is client-side (`external_key` + query-before-write + local ledger). Notion version history is plan-limited (7 days Free), so undo relies on our own write ledger (archive the page/block we created), which only works for append-only writes into Scrollback-owned surfaces.

## 2026-07-10 — Open core under FSL-1.1-MIT + GitHub artifact attestations; Free + Pro $15/mo; Paddle/LS payments; no accounts, no backend, no telemetry
**What:** `scrollbackd` + `scrollback-mcp` source-available under FSL-1.1 (converts to MIT after 2 years); app shell + filing agents + recipes proprietary. Offline-verifiable Ed25519 license keys; egress ledger as a first-class UI surface.
**Why:** Auditability is the conversion lever for an always-on capture daemon (Hyprnote/Bitwarden pattern); plain MIT invited the fork that Screenpipe blamed for its model ("open source on its own is not a business model"); no backend = nothing to breach (contrast Cluely's 83K-user leak) and ~$140/yr fixed costs.
**Rejected:** MIT core (fork-exposed), fully-proprietary + audit-only (forfeits the verifiability moat vs minimi), subscription-with-account infrastructure (adds a breachable backend).

## 2026-07-11 — Filing runner defaults to the user's Claude subscription
**What:** Default `FilingRunner` = user's existing Claude subscription (Agent SDK / `claude -p` / Claude Desktop scheduled task), $0 marginal cost, no stored LLM credential. Anthropic API-key mode (Haiku-tier, ≈$1.6–7/mo, with a live cost display) is the built-in swappable fallback; local-LLM mode is the offline option.
**Why:** Founder decision — zero added user cost with identical model quality; the API-key fallback is one setting away if plan limits bite or if Anthropic lands its paused-but-signaled split of SDK billing from subscriptions.
**Rejected:** API key as default (adds recurring cost for the common case), local-LLM as default (real accuracy drop on extraction/summarization).
**Gotcha:** the Desktop-scheduled-task variant needs the app open + machine awake; prefer the Agent SDK / headless CLI route as the primary within this default (more controllable, no foreground dependency).

## 2026-07-11 — Storage integration: system SQLite now, SQLCipher before dogfood (user-approved)
**What:** Build the catalog store on the OS's built-in SQLite (`import SQLite3`, FTS5 included) immediately — schema, `user_version` forward-only migrations, and DAOs — with every `sqlite3_*` call isolated in one thin wrapper (`SQLiteDatabase`) and the encryption seam at a single spot (`PRAGMA key` after open). Turn on **SQLCipher encryption in store (b)**, before any real founder data persists; the daemon is NOT wired to the store as its live sink until then (still the throwaway JSONL spike).
**Why:** SQLCipher is **API-identical** to SQLite — an encrypted DB is the same DB plus a key — so switching is (1) link SQLCipher instead of system libsqlite3 and (2) issue `PRAGMA key` first; DAOs are unchanged. That makes "build now, encrypt before real data" near-zero rework, not a rewrite, so it preserves momentum while still guaranteeing encryption before the honeypot exists (zero real data until the M1 gate run). Presented three options to the user (vendor amalgamation / GRDB+SQLCipher / build-now-encrypt-before-dogfood) in plain language; the user chose the phased path after understanding the "same DB + a key" equivalence.
**Rejected:** GRDB.swift + SQLCipher (a sizable third-party Swift dependency inside the trust-critical daemon — bigger audit surface, against the minimal-dependency/verifiable-core story); vendoring the SQLCipher amalgamation *up front* (a day of build-flag/crypto-provider yak-shave blocking momentum now, and it needs downloading source — deferred, not rejected: it remains the likely eventual destination for the audit-surface story once the store stabilizes).
**If you change it:** keep all `sqlite3_*` calls behind `SQLiteDatabase` and the key application at the single `PRAGMA key` seam — that isolation is exactly what makes the SQLCipher swap cheap. `chunks_vec` (sqlite-vec `vec0`) is deliberately NOT in migration v1 (needs the extension) — it arrives as migration 2 with the embedding task.

## 2026-07-11 — Default-on exclusions: pattern rules (core) + runtime OS signals (daemon)
**What:** Two-layer capture exclusion. (1) **Pattern rules** — a pure `ExclusionSet` (ScrollbackCore): `ExclusionRule {type: app|url|window|regex|schedule, pattern, mode: neverCapture|redact}`, resolving a `FrontmostContext` to the strictest matching `CaptureMode` (`neverCapture` > `redact` > `capture`). Honored in `CaptureEngine`: `neverCapture` opens no episode and stores nothing (gated in `handleAppOrWindowChange` before the same-key branch, plus `handleClipboard`/`noteUserActivity`/`capture`); `redact` records the episode but stores a fixed placeholder instead of content and never reads the provider. Defaults: password-manager bundle IDs, the Claude Desktop bundle id, incognito/private window-title markers. (2) **Runtime OS-state signals** — `CaptureGuards` (scrollbackd): `IsSecureEventInputEnabled()` and `NSWindowSharingNone` (via `CGWindowListCopyWindowInfo`), gating the live AX + OCR providers (return nil when suppressed).
**Why:** PRD:102 / CLAUDE.md Never-rule require these honored + advertised. The pattern/OS split mirrors the redact-stage split: data-driven rules live in the pure, headless-testable core (so the never-capture invariant has real regression tests — a broken rule fails CI, not by leaking a password in production), while OS-state signals that need process/window APIs live in the daemon (observed manually, like the rest of the live capture path). Gating `neverCapture` at episode-open (not just at emit) means an excluded app produces no episode metadata at all, satisfying the DoD "excluded apps provably absent from the store."
**Rejected:** exclusion checks only at emit (would still record excluded episodes' metadata — app/window/time — of a password manager); a hardcoded banking bundle-id list (arbitrary + incomplete; banking is mostly web → left to user URL rules + a future default URL set); putting the OS signals in the core (needs daemon-only APIs, untestable there anyway). Incognito detection is best-effort by window title (a browser doesn't always surface private mode in the AX title) — documented as such, not a guarantee.
**If you change it:** default rules must never match the `simulate`/test fixtures (`com.apple.Safari`, `com.tinyspeck.slackmacgap`) or every golden count breaks — there's a `testDefaultsDoNotExcludeOrdinaryApps` guard. The four engine gates (app/window change, clipboard, resume-after-idle, capture) are all load-bearing; removing any one reintroduces a capture path for excluded apps.
**Hardened by a 4-lens adversarial review (4 findings, all fixed; no gating gap — the never-capture invariant was verified intact):** (1) the `NSWindowSharingNone` guard inspected the app's *topmost* window while the AX walker reads the *focused* window (and its `layer==0` filter skipped sharing-none windows at other levels) → replaced with a **fail-safe**: suppress the whole pid if *any* on-screen window it owns opts out. (2) redact mode stored the verbatim `windowTitle` in episode metadata — for a `.window` rule the title *is* the sensitive datum → `openEpisode` now masks the title for redact apps (app still identifiable by bundleID/appName). (3) `.url`/`.schedule` rule types are **inert** in the live pipeline (no URL/clock plumbed) — documented honestly + a browser-URL-extraction TODO added; no default rule uses them, so shipped protections are unaffected. (4) the redact-mode *clipboard* path was untested → added.

## 2026-07-11 — Redact stage: secrets-only, high-precision, at the emit chokepoint (defense-in-depth)
**What:** Capture-time `Redactor` (pure, in ScrollbackCore) masks high-risk secrets — PEM private keys; `sk-`/`gh_`/`AKIA`/`AIza`/`xox-`/JWT tokens; and Luhn-validated card numbers — with `[redacted:<name>]`, setting an `events.redaction_flags` bitmask (`RedactionFlags`). It is applied in `CaptureEngine.emit`, the single chokepoint every captured span (screen, clipboard, future audio) passes through, and the dedup hash is taken over the REDACTED text so a secret never enters the hash. **PII (emails, phone numbers, names) is deliberately NOT redacted.** Rules are extensible (a user/exclusion rule contributes `.custom`).
**Why:** The specs frame redaction as **defense-in-depth, not a perfect-redaction promise** (PRD:147 — "Recall proves NPU-grade filters miss unlabeled secrets"; the real guarantee is on-device encryption). The product's *value* is remembering who/what — emails, names, and context are the signal, so scrubbing PII would gut it. So the default is deliberately narrow and high-precision: only the "must-never-persist" secret category, every pattern anchored to a recognizable secret shape (specific prefixes, PEM markers, or Luhn-valid PANs) to avoid masking ordinary text. Placing redaction at `emit` (not per-source) means no capture path can ever store an un-redacted secret.
**Rejected:** PII redaction by default (destroys the ambient-memory value; emails/names are the point), flag-only-without-masking (stores the secret, only tagged — defeats the purpose), redaction as a "guarantee" (over-promises; classifiers/regex miss unlabeled secrets — PRD is explicit), per-source redaction (a new capture path could bypass it).
**If you change it (add PII categories, or make it a stronger promise):** reversible today — the spike JSONL store is throwaway and the encrypted store isn't built yet, so a scope change only affects future captures. Revisit before the encrypted store lands, and never advertise it as more than defense-in-depth. Adding a rule = one `RedactionRule` in `Redactor.defaultRules` + a `RedactionFlags` bit + a test.
**Hardened by a 4-lens adversarial review (7 findings, all fixed):** (1) **ReDoS** — the private-key regex's `[\s\S]*?` scanned to EOF for every `BEGIN`, so an on-screen `-----BEGIN…PRIVATE KEY-----` flood with no `END` (attacker-controlled ambient text) was O(n²) = an 8–18 s main-loop freeze / remote capture-DoS; fixed by bounding the body (`{0,8192}?`) + header classes and a cheap `guardSubstrings` `contains` pre-gate. (2) **Clipboard leak** — clipboard stored *verbatim* while the hash pass redacted the *normalized* form, so a card with tab/newline/NBSP separators leaked the full PAN into `rawText` with empty flags; fixed by **storing the normalized clipboard form** (verbatim clipboard storage dropped — no paste-back feature depends on it) and collapsing `emit` to **one redaction pass** that drives rawText + hash + flags together (they can no longer disagree). (3) **Card over-match** — a card adjacent to a stray digit formed a 17-digit run that failed Luhn as a whole and leaked; fixed with `isRedactableCardRun` (standard-length runs Luhn as-is; 17/18-digit runs check standard-length prefixes/suffixes).

## 2026-07-11 — OCR fallback: sync seam, per-app capability matrix, zero-frame, ScreenCaptureKit+Vision
**What:** Apple Vision OCR fallback for AX-opaque surfaces, composed behind the existing *synchronous* `TextSnapshotProvider` seam by `LayeredTextSnapshotProvider`. Routing is a per-app capability matrix (`AppCaptureCapabilities` → `CaptureStrategy` `axOnly | axThenOCR | ocrOnly`) with a **conservative `axThenOCR` default**: OCR fires only when AX text is empty/thin (`OCRFallbackPolicy`, ≤16 normalized chars), never regressing below AX (prefer whichever produced more text). Live path (`VisionOCRExtractor`, scrollbackd): ScreenCaptureKit window screenshot → Vision `VNRecognizeTextRequest` → `OCRTextAssembler` reading-order join; the `CGImage` feeds Vision and is discarded immediately (zero frames stored). Requires the Screen Recording grant; absent it, OCR returns nil and capture degrades to AX-only. New `scrollbackd ocr-dump` diagnostic mirrors `ax-dump` for the granted gate run.
**Why:** OCR is genuinely async (ScreenCaptureKit), but making the seam async would ripple through the engine and destroy its deterministic headless drive (`simulate` + XCTest). Keeping the seam sync and bridging the *rare* screenshot async→sync at the scrollbackd boundary preserves the verifiable engine. The `axThenOCR` default satisfies the DoD ("an unknown Electron/canvas app with empty AX still produces text") while a rich-AX native app never pays the OCR/CPU cost — holding the text-first, zero-frames, <5% CPU promise. macOS 15 floor makes ScreenCaptureKit (not deprecated `CGWindowListCreateImage`) the right capture API.
**Rejected:** async provider seam (kills the deterministic engine — the whole headless-verifiability hedge); `axOnly` default (a novel AX-opaque app would silently capture nothing — fails the DoD); always-OCR / OCR-every-frame (CPU + privacy cost, defeats text-first); `CGWindowListCreateImage` (deprecated on our floor).
**Gotcha — OCR bypasses the AX secure-field guard (partially mitigated in-increment):** the AX walker skips secure fields (`AXCapturePolicy.isSecureField`), but a *screenshot* captures the whole window with no field-level redaction — inherent to pixel OCR (every analog — Rewind/Screenpipe — has it). **Mitigation added here** (from the review): the AX seam now carries `CapturedText.containedSecureField`, set when the walker skips a secure field, and `LayeredTextSnapshotProvider` refuses OCR for that window — so a normal credential window (whose AX exposes chrome/labels + a skipped secure field) is never screenshotted. **Residual gap** (still deferred to the NEXT tasks): an all-canvas login whose AX is *entirely* empty carries no signal, and `ocrOnly` apps skip the AX probe — both closed by app-level **default exclusions** (password managers/banking/incognito never captured — Never-rule) + the **redact stage**. A future enhancement can mask AX-known secure-field rects out of the image before OCR. Must not ship to dogfood before default exclusions land.
**If you change it:** the sync-seam decision is load-bearing for `simulate`/XCTest determinism — an async seam forces the engine async and breaks every headless gate.

## 2026-07-11 — Near-dup collapse: MinHash + LSH banding, two-stage gate, pre-embedding
**What:** A pure `MinHasher` + streaming `NearDupCollapser` (ScrollbackCore) that drops chunks which are *near* (not byte-equal) duplicates of already-retained text — the scrolled or lightly-edited screen re-read that exact-hash dedup misses. MinHash signature = 128 affine permutations (`(a·x+b) mod (2³²−5)`) over word-trigram shingles hashed with FNV-1a-32; Jaccard is estimated as the fraction of matching signature positions. LSH banding (16 bands × 8 rows) surfaces candidates; a **second exact signature-Jaccard check enforces the 0.85 threshold** before any collapse. Wired into `ChunkingStage` *after* the cheap exact-hash miss; a collapse increments `nearDupSkips` and drops the chunk (default on; `nearDup: nil` disables). Coefficients are derived once from a fixed SplitMix64 seed → signatures are byte-identical across runs/machines (a corpus dedups the same way every launch).
**Why:** Embedding + the vector index are the expensive resources; a re-read of a scrolling document is ~identical text that would double-charge both. Exact-hash only collapses byte-equal re-reads. MinHash/LSH is the standard sub-quadratic near-dup primitive (a new chunk compares only against LSH-band candidates, not the whole corpus). The **two-stage gate** (permissive banding → exact 0.85 verify) is deliberate: banding's S-curve inflection (≈0.71 at 16×8) sits *below* the threshold so genuine near-dups aren't missed by the pre-filter, then the signature check enforces the real 0.85 so distinct content isn't silently dropped. On a mixed repetitive stream (10 distinct docs, 15 scrolled re-reads) it collapses 60% of offered chunks.
**Rejected:** pairwise Jaccard against every retained chunk (O(n²) — unbounded as a session grows); banding-only with no exact verify (the S-curve is soft — pairs well below 0.85 would occasionally collapse, silently dropping distinct memory); a random-seeded permutation family (non-reproducible signatures — the same corpus would dedup differently each launch, and tests couldn't assert exact behavior); 64-bit/2⁶¹-prime hashing (needs 128-bit mul-mod; the 32-bit domain with prime 2³²−5 keeps `a·x+b` provably < 2⁶⁴ so plain `&*`/`&+` suffice).
**If you change it:** false collapses drop real memory content — keep the exact signature-Jaccard verify after banding; never collapse on band-candidacy alone. First-seen wins as the representative; **AX-over-OCR preference on collapse is a TODO** (needs the representative's `CaptureSource` threaded through — an OCR-noisy chunk should yield to a later AX-clean near-dup). The in-memory signature index is bounded by a session's chunk count for the spike; once the store lands, the retained-signature set moves behind the store like the exact-hash index.

## 2026-07-11 — Dual timestamps: ts_event extraction resolved against CAPTURE time (not now)
**What:** A pure `EventTimeExtractor` (ScrollbackCore) fills `chunks.ts_event` — the event time a chunk *refers to* — distinct from `ts_capture` (when it was seen). It resolves a curated, high-precision pattern set relative to the CAPTURE timestamp: ISO dates (`2026-07-15`), month-name dates (`Jul 15`, `15 July`, optional year), relative day words (`today`/`tomorrow`/`yesterday`), and weekdays with `next`/`this`/`last` qualifiers (`Friday` said Monday → that week's Friday). The FIRST recognizable reference in reading order wins (absolute beats relative on a location tie); wired into `Chunker.chunk` so every chunk carries both timestamps. Injected calendar/timezone → deterministic (tests pin UTC; the live daemon uses the user's zone).
**Why:** The dual-timestamp requirement — "meeting moved to Friday" captured Monday must be retrievable by BOTH Monday and Friday. Relative expressions MUST resolve against when the text was captured, not wall-clock now (a note read weeks later still means the Friday after *its* Monday). `NSDataDetector` resolves relative dates against the current date — both semantically wrong here and non-deterministic (breaks headless verification) — so a focused custom resolver is the right call. Precision over recall, like the redactor: only high-confidence date shapes fire, impossible dates (`2026-13-45`, `Feb 30`) are round-trip-rejected, and bare numbers/versions/ports never match — a missed date beats a wrong `ts_event` polluting time-filtered recall.
**Rejected:** `NSDataDetector` (now-relative + non-deterministic — see above); resolving relative to wall-clock now (wrong for any non-live re-read); storing multiple event times per chunk (needs an `event_times` child table — deferred; one event time covers the DoD); combining date+time-of-day (`3pm on the 15th`) and standalone times (`call at 3pm`) — deferred as lower-signal, higher-false-positive surface.
**If you change it:** keep resolution relative to `capturedAt`, never `Date()`. Known limitation: `may` is both a month and a modal verb, so `you may 5 times` can false-match May 5 — accepted for a soft retrieval signal (not a security gate; a rare mis-dated chunk is recoverable), documented rather than special-cased. `next`/`last` weekday follow a documented convention (next-week's / most-recent-past occurrence) since natural language is genuinely ambiguous. Adding a pattern = one regex + a resolver branch + a precision test proving it doesn't fire on ordinary text.

## 2026-07-11 — Hybrid retrieval: pure ranking policy + SQL candidate generation, RRF over FTS5 + recency
**What:** Hybrid recall split into (1) a pure `HybridRanker` (ScrollbackCore) that RRF-fuses N ranked candidate lists then diversifies by episode (soft per-episode cap with backfill), and (2) `SQLiteCatalogStore.hybridSearch` that generates the candidate lists in SQL — a BM25 keyword list (`chunks_fts MATCH`) and a recency list (`ORDER BY ts_capture DESC`) — under hard time/app/entity pre-filters, fuses via the ranker, joins `events` for provenance, and returns `[SearchResult]`. Vector similarity slots in later as a THIRD fused list with no change to the fusion/diversification policy. Free-text is sanitized into a safe FTS5 `"tok" OR "tok"` expression (raw `MATCH` would AND every term → near-zero recall, and throws on punctuation); entity filters use `LIKE ? ESCAPE '\'` against the JSON `entity_keys`; the caller `limit` is clamped (see review below).
**Why:** The PRD DoD needs both "what did I do today?" (time-scoped browse — carried by the recency list when keywords don't match the text) and a one-episode-dominant query returning diversified episodes (the per-episode cap). Splitting the *policy* (pure, DB-free) from *candidate generation* (SQL) is what lets the diversification/fusion DoD be asserted deterministically in a unit test while the SQL path is exercised end-to-end against an in-memory DB. RRF avoids calibrating BM25 vs recency vs (future) cosine onto a common scale. Provenance is carried from the joined event and fail-safes to `.untrustedAmbient` if ever unknown — an untrusted-ambient span must never be mislabelled trusted before it reaches the MCP spotlight.
**Rejected:** ranking inside SQL (couldn't unit-test the diversification policy without a DB; and RRF across heterogeneous lists is awkward in SQL); raw-text FTS `MATCH` (AND-of-terms brittleness + syntax errors on punctuation); making `SQLiteCatalogStore` conform to `RetrievalStore` (async, `Sendable`) now (it holds a raw sqlite3 pointer and is synchronous-by-contract — the async actor wrapper lands with live wiring; conforming now would be a Sendable lie); a hard per-episode cap with no backfill (a few-episode corpus would return fewer than `limit` results).
**Hardened by a bounded 4-lens adversarial review (correctness / SQL-safety / security-spec / resources-concurrency), each finding independently refuted → 1 confirmed of 2 raw:** caller-supplied `query.limit` was unclamped, so `max(query.limit * 5, 50)` could trap on Int-overflow (a latent daemon crash on the public API) and a `limit` > SQLite's bind-variable cap would overflow the `IN(...)` list. Fixed by clamping at the retrieval boundary (`min(max(limit,0), maxResultLimit=200)`) with overflow-safe arithmetic + a `limit: .max` regression test. The reviewers explicitly cleared: SQL parameterization (no interpolation of user text), FTS/LIKE escaping, provenance carry + fail-safe, pre-filter exclusion on BOTH candidate queries, statement finalization via `defer`, and that no false `Sendable`/`RetrievalStore` conformance is claimed.
**If you change it:** keep candidate generation (SQL) separate from the ranking policy (pure) — that seam is the testability hedge. Vector search must enter as another `rankedLists` entry, not a reranker bolted onto SQL. Never interpolate query text into SQL; keep the FTS sanitizer + `LIKE ESCAPE`. Provenance must stay fail-safe to `.untrustedAmbient`.

## 2026-07-11 — ScrollbackBench retrieval-quality net + stopword sanitization (bench-driven fix)
**What:** A `swift test`-wired retrieval-quality regression net (`ScrollbackBenchTests`): seed a small varied corpus of labelled episodes, run a golden set of natural-language "what was that thing I saw?" probes, and assert the right episode lands in the top-3 (recall@3 == 1.0), plus a time-scoped probe. ## 2026-07-11 — Weekly-shard store: one encrypted SQLite file per ISO week, purge = file delete
**What:** Storage is a directory of per-ISO-week `SQLiteCatalogStore` files (`scrollback-2026-W28.sqlite`), managed by `ShardedCatalog`. Writes are episode-atomic — a whole episode (its events + chunks) lands in the shard of its `tsStart` week, so foreign keys never cross files. Reads fan out across the shards a query's `timeRange` touches (all shards if unscoped) and fuse the per-shard `[SearchResult]` orderings via `HybridRanker` (a single-shard query bypasses re-fusion and equals querying that shard directly). `purge(before:)` deletes the whole shard files (+ `-wal`/`-shm` sidecars) whose ISO week is entirely before the cutoff, returning the dropped shards. Pure ISO-week math (`WeekShard` + `WeekShardCalendar`) does the routing; SQLCipher is a pass-through `key` handed to each shard store.
**Why:** The privacy claim (CLAUDE.md / PRD) is that "delete everything before X" is INSTANT and PROVABLE — a file unlink, not a row-by-row DELETE you can't prove completed. Per-week files make that literal: the erased weeks' files are gone from the directory. Episode-atomic placement makes a whole episode a single unit of retention (no orphaned events after a purge). Fanning out + fusing across shards reuses the RRF ranker already built, so cross-shard search cost ~ the small number of weeks a time-scoped query touches (usually 1–2). Presented in plain language + recommended; the user said "keep building" so it was chosen and logged here.
**Rejected:** one big DB + a `week` column, purge = `DELETE WHERE week < X` (exactly the row-DELETE the PRD rejects — slow, and you can't PROVE the rows/pages are unrecoverable); one catalog + sharded vector/blob satellites only (purge still row-DELETEs the shared catalog — same problem); global cross-shard RRF over raw candidate lists (needs per-shard metadata refetch after fusion — more coupling; re-fusing self-contained per-shard result lists is simpler and identical for the common single-shard case). Purge granularity is the WEEK: an episode straddling the cutoff (started before X, ran past it) is dropped with its start-week shard — documented, acceptable for the week-granular contract.
**If you change it:** keep episode-atomic placement (splitting an episode across shards breaks the chunk→event/episode joins retrieval needs). The async `RetrievalStore` conformance is a thin ACTOR wrapper over this sync manager (the actor is the "confine to one queue" boundary the per-shard stores require) — add it at live wiring, along with the upsert-granularity question (the protocol's `upsert(chunks)` vs. this episode-atomic `ingest`). Encryption stays a per-shard `key` pass-through.

## 2026-07-11 — Filing quarantine validator: schema-only + no-URL exfil gate (scan DECODED values)

**If you change it:** the DECODED-value scan is the authoritative exfil gate — never move it back to raw-only. Keep it fail-closed on every branch. This is the write-side twin of MCP spotlighting (read-side): together with the redact stage they are the three ambient-as-data boundaries.

## 2026-07-11 — MCP daemon socket transport (POSIX AF_UNIX; the live edge, review-hardened)
**What:** The transport that carries recall from the daemon to the `scrollback-mcp` proxy (and thus Claude). Split by verifiability. **Pure/headless (Core):** `MCPFraming` (4-byte big-endian length prefix + JSON body) + `MCPFrameAccumulator` (streams → complete frames, handling partial/coalesced reads, enforcing a **1 MiB cap on the length prefix BEFORE buffering** so a hostile header can't OOM the daemon); `MCPToken` (128-bit system-random, **constant-time** compare, hex); `MCPWireProtocol` — `MCPConnectionHandler`, a per-connection **handshake state machine**: the FIRST frame must be a valid `hello` with the correct token; any pre-auth misstep (wrong token / gated method before hello / undecodable frame) replies then CLOSES (one-shot handshake, bounding brute force to one guess per connect); post-auth a malformed frame is tolerated. Two error planes: **transport** (`UNAUTHORIZED`/`NOT_AUTHENTICATED`/`MALFORMED`/`UNKNOWN_METHOD`) vs **application** (the existing `LOCKED`/`RATE_LIMITED`/… inside the tool result). **Live (scrollbackd):** `MCPSocketServer` — `socket(AF_UNIX, SOCK_STREAM)` at `~/Library/Application Support/Scrollback/mcp.sock` (mode 0600), token in a mode-0600 file created owner-only, one accept thread + one thread per connection, every `handle` funneled through a single serial `requestQueue` (the shared `MemoryMCPService` + `QueryThrottle` are single-threaded by contract). New `scrollbackd mcp-serve` command serves recall with NO TCC and NO networking (reads the store) so the transport is exercisable independently of a capture session.
**Why:** The product's delivery mechanism is "serve memory to Claude over MCP," and the daemon must hold key custody + the spotlight boundary while the `.mcpb` proxy stays a dumb relay (tech-spec §3a/§3b). POSIX `AF_UNIX` is deliberately NOT Network.framework: a Unix domain socket is local IPC (no packets leave the machine), and `NWListener`/`CFSocket` would trip the no-networking grep gate AND blur the local-IPC/egress line that IS the privacy claim. `socket()`/`bind()`/`accept()` are correctly absent from the grep denylist (only `CFSocket`/`SocketPort`/`NW*`/`URLSession`/`getaddrinfo`/`CFStream` are). Building the protocol as a pure clock-injected state machine means auth-gating, the DoS frame cap, and the two error planes are all proven headless (`MCPTransportTests`, +21 tests); only the syscall edge needs the Mac.
**Reviewed:** a 5-lens adversarial workflow (auth-bypass, DoS, concurrency, framing, privacy) + per-finding verification confirmed 5 real defects — **ALL in the live socket layer; the pure framing/token/handshake logic held up**. Fixes: (1) **SIGPIPE (was the top finding — a same-UID client sends one frame then closes its read half → `write()` raises SIGPIPE → the WHOLE daemon dies, capture+indexer+key-custody, not just the connection; verifier reproduced exit 141):** `signal(SIGPIPE, SIG_IGN)` backstop at `start()` + per-accepted-fd `SO_NOSIGPIPE` (not inherited across `accept`), so a dead peer yields `EPIPE` and drops only that connection. (2) **Unauthenticated slowloris (no token needed to connect — a silent peer pins a slot forever; 8 slots → recall denied):** an `SO_RCVTIMEO` handshake deadline reaps a peer that hasn't completed `hello`, relaxed to blocking once authenticated (an authed peer already holds the token → full read access, so pinning a slot buys nothing). (3) **accept() treated transient errors as fatal** (one `ECONNABORTED`/`EMFILE` permanently killed the recall socket): now continues on transient errors, backs off on resource pressure, returns only on `stop()` or a fatal listener error.
**Verified LIVE (`scripts/mcp-probe.py` against the real captured store):** file perms 0600; both auth gates return the right code AND close the connection; happy path hello→tools/list(2 read-only tools)→tools/call returned 3 real spotlighted snippets; transport-vs-application error planes; and the **SIGPIPE fix confirmed — the daemon survived 5 dead-peer writes and kept serving**. Kept as the reusable manual gate (verify skill step 4) — the only proof for the un-headless-testable edge.
**Rejected:** Network.framework/`NWListener` (trips the grep gate; blurs local-IPC vs egress); a fresh token per connection or in-band key exchange (the mode-0600 token FILE minted at first-run is the tech-spec §4 design; Keychain custody lands with the SE key); GCD queues for blocking accept/read (starves the pool — dedicated threads + one serial request queue is correct for single-digit local connections); dropping authenticated idle connections on timeout (would force the persistent proxy to reconnect and buys no security — an authed peer already has the token).
**If you change it:** ALL service access stays on the single serial `requestQueue` (the service + throttle are not thread-safe). Per-fd `SO_NOSIGPIPE` is mandatory on every accepted socket (not inherited across `accept`) — plus the global `SIG_IGN` backstop. The handshake `SO_RCVTIMEO` must stay (the unauth-slowloris fix). `mcp-serve` currently runs with `isLocked=false` (plaintext walking skeleton); wire the real Secure-Enclave lock gate here when key custody lands. Standalone this increment — folding the same server into `runDaemon()` (serve-while-capturing) is the follow-up; the **Node proxy (`scrollback-mcp`)** that bridges MCP stdio ↔ this socket is the next increment (it reads the token file, frames per this protocol, translates `tools/list`/`tools/call`).

## 2026-07-11 — scrollback-mcp: the thin Node proxy (recall path completed end-to-end)
**What:** The `.mcpb` stdio proxy that bridges an MCP client (Claude Desktop/Code) to the daemon recall socket — the last mile of the recall path. `scrollback-mcp/index.js` speaks newline-delimited MCP JSON-RPC on stdio (`initialize`/`tools/list`/`tools/call`/`ping` + ignores `notifications/*`) and, per call, opens a short-lived AF_UNIX connection to the daemon, completes the token handshake, forwards the request with the daemon's length-prefixed wire framing, and translates the reply back: daemon tool defs → MCP `tools/list` (with `annotations.readOnlyHint`); daemon `MCPCallResponse` → MCP `content:[{type:text}]`. **Zero npm dependencies** (Node built-ins only — `net`/`fs`/`readline`): the proxy holds no memory, no DB key, no retrieval/spotlight logic (tech-spec §3a "thin proxy; key custody never leaves the daemon"), and there's no supply-chain surface. Also a one-line supporting change in the DAEMON: `MCPResultFormatter.Response.rendered` went from a COMPUTED property to a STORED, wire-carried field (assembled once in `format()` via `MCPResultFormatter.render`) so the proxy forwards the daemon's already-spotlighted text VERBATIM — the unforgeable-fence assembly never moves into the thin proxy. `manifest.json` (`.mcpb`), `package.json` (no deps), `README.md` (Claude Desktop config), `smoke.mjs` (the live proxy gate).
**Why:** The product's whole delivery mechanism is "serve memory to Claude over MCP," and that requires a real MCP endpoint. Keeping the proxy dumb (translate + forward only) is the security posture: every decision that matters — auth, throttle, lock, spotlighting, key custody — is already enforced in the daemon, so the forgettable Node layer can't weaken them. Implementing MCP by hand (no `@modelcontextprotocol/sdk`) keeps it thin AND dependency-free (also unblocked in this offline env). Short-lived connection-per-call makes the proxy stateless + restart-safe (re-reads the token each call) and sidesteps stale-connection handling; the local handshake is cheap.
**Verified LIVE (`scrollback-mcp/smoke.mjs` against `scrollbackd mcp-serve` + the real store):** 10/10 — `initialize` returns serverInfo + tools capability; `tools/list` → both read-only tools each with a JSON-Schema `inputSchema`; `tools/call search_memory` → text content, not isError; missing-query → `isError` with the INVALID message (application error plane); `recent_activity today` → text content; unknown method → JSON-RPC `-32601`. The full path (client ↔ proxy stdio ↔ daemon socket ↔ recall service ↔ store) works end to end.
**Rejected:** the `@modelcontextprotocol/sdk` dependency (adds a dep to the trust-critical delivery path + needs a network install; MCP-over-stdio is small enough to hand-implement); reconstructing the rendered/spotlighted text in the proxy (moves the injection-fence assembly into the thin layer — instead the daemon now sends `rendered` on the wire); a persistent daemon connection (needs stale/restart handling for no benefit over cheap local per-call connects); putting tool schemas in the proxy (the daemon owns the contract; the proxy relays `tools/list`).
**If you change it:** the proxy stays dumb — no memory, no key, no retrieval/spotlight logic; it only translates + forwards. The daemon's `Response.rendered` MUST stay wire-carried (the proxy forwards it verbatim; never re-derive fences downstream). `smoke.mjs` needs a running `mcp-serve` — it's the only proof of the stdio↔socket bridge (the Swift side can't test the Node layer). Full end-to-end via actual Claude Desktop still needs the founder's machine (config the `.mcpb`); the smoke test covers everything up to that wiring.
